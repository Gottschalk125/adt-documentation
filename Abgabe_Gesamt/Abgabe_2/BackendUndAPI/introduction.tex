\section{Backend und API}
\subsection{Technologieentscheidung (5123097)}

Die Architektur des Backends sowie die Gestaltung der Schnittstellen sind entscheidend für die Wartbarkeit, Sicherheit und Skalierbarkeit des Hospital Management Systems. 
Für dieses Projekt wurde eine Kombination aus der Programmiersprache \textbf{Kotlin} und dem Architekturstil \textbf{REST} gewählt.

\subsubsection{Programmiersprache: Kotlin}
Obwohl Java der traditionelle Standard im Enterprise-Umfeld ist, bietet Kotlin moderne Sprachfeatures, die insbesondere in einem sensiblen Bereich wie dem Gesundheitswesen von Vorteil sind:

\begin{itemize}
    \item \textbf{Null-Safety (Typsicherheit):} Einer der häufigsten Fehler in der Softwareentwicklung sind \texttt{NullPointerExceptions}. 
    Kotlin integriert die Prüfung auf Null-Werte direkt in das Typsystem. 
    Da medizinische Daten (z. B. Allergien oder Vorerkrankungen) oft optional sind, zwingt Kotlin den Entwickler dazu, diese Fälle explizit zu behandeln, was die Systemstabilität massiv erhöht. \\
    Durch diesen Aspekt wurde auch bewusst auf eine Umsetzung mit Python in Kombination mit der FastAPI verzichtet, da Python dynamisch typisiert ist. 
    Fehler bei der Datentyp-Zuweisung (z.B. ein String, bei dem eine Zahl für eine Medikamentendosis erwartet wird) fallen oft erst zur Laufzeit beim Endnutzer auf. 
    Kotlin hingegen ist statisch typisiert, sodass der Compiler vorab die Datenflüsse validiert. Dies ist in einem Hospital Management System essenziell, um lebenskritische Datenfehler zu minimieren, wodurch auf eine komfortablere Entwicklung in Python bewusst verzichtet wurde.
    \item \textbf{Prägnanz und Wartbarkeit:} Durch Features wie \textit{Data Classes}, \textit{Extension Functions} und \textit{Type Inference} reduziert Kotlin den Boilerplate-Code deutlich im Vergleich zu Java. Dies macht den Code lesbarer und weniger anfällig für Implementierungsfehler in der Geschäftslogik.
    \item \textbf{Volle Interoperabilität:} Kotlin läuft auf der Java Virtual Machine (JVM) und ist weitestgehend mit Java kompatibel. Dies erlaubt die Nutzung von etablierten Frameworks wie \textit{Spring Boot}, das für die Anbindung der PostgreSQL-Datenbank und die Absicherung des Systems essenziell ist.
\end{itemize}

\subsubsection{Schnittstellenarchitektur: REST-API}
Für die Kommunikation zwischen dem Backend und den Client-Anwendungen wurde das Paradigma des \textbf{Representational State Transfer (REST)} gewählt aufgrund folgender Überlegungen:

\begin{itemize}
    \item \textbf{Zustandslosigkeit (Statelessness):} Jede REST-Anfrage enthält alle Informationen, die für die Verarbeitung notwendig sind. Dies vereinfacht die Skalierbarkeit des Servers und verbessert die Zuverlässigkeit, da keine komplexen Session-Zustände auf dem Server verwaltet werden müssen.
    \item \textbf{Plattformunabhängigkeit:} Die API liefert Daten standardmäßig im JSON-Format. Dies ermöglicht eine strikte Trennung von Geschäftslogik und Benutzeroberfläche. So kann dasselbe Backend problemlos ein Web-Dashboard für die Verwaltung, eine mobile App für Ärzte oder externe Laborsysteme bedienen und ermöglicht somit effiziente Erweiterungsmöglichkeiten des Projekts.
    \item \textbf{Standardisierte Kommunikation:} Durch die Nutzung der HTTP-Verben (\texttt{GET} zum Abrufen von Patientenakten, \texttt{POST} für Neuaufnahmen und Änderungen) wird eine intuitive und konsistente Schnittstelle geschaffen, die sich bereits etabliert hat.
    %TODO wirklich Caching mit reinbringen? nutzen wir das?
    %\item \textbf{Caching-Effizienz:} Medizinische Stammdaten, wie Medikamentenlisten, ändern sich selten. REST unterstützt natives HTTP-Caching, wodurch die Netzwerklast reduziert und die Antwortzeiten für das medizinische Personal optimiert werden.
\end{itemize}

Neben REST wurden auch modernere Ansätze wie GraphQL und gRPC in Betracht gezogen. Basierend auf den Anforderungen des Systems wurde sich jedoch bewusst für eine REST-API entschieden:

\begin{itemize}
    \item \textbf{REST vs. GraphQL:} 
    Während GraphQL es Clients ermöglicht, passgenaue Datenstrukturen abzufragen, erhöht es die Komplexität im Backend (Query-Parsing, N+1-Problematik) und erschwert effizientes Caching auf HTTP-Ebene. 
    Da die Datenmodelle im Krankenhauswesen in der Regel klar definiert und standardisiert sind, bietet REST eine robustere Performance und ist einfacher abzusichern, da Zugriffsrechte auf Endpunkt-Ebene definiert werden können.
    
    \item \textbf{REST vs. gRPC:} 
    gRPC bietet zwar durch die Verwendung von Protocol Buffers eine höhere Performance und geringere Latenz, erfordert aber spezialisierte Client-Bibliotheken und ist für Browser-Anwendungen schwerer zu integrieren, da beispielsweise Proxys wie gRPC-Web benötigt werden.
    REST hingegen nutzt Standard-JSON über HTTP/1.1 oder HTTP/2, was die Integration von Web-Frontends und Drittanbietersystemen ohne zusätzliche Komplexität sicherstellt.
\end{itemize}

\textbf{Design der Endpunkte}

Bei der Gestaltung der REST-Endpunkte wurde streng auf Konformität und Robustheit geachtet:

\begin{itemize}
    \item \textbf{Explizite Fehlerzustände:} Anstatt bei nicht vorhandenen Ressourcen \texttt{null} oder leere Listen zurückzugeben, wirft das Backend konsequent eine \texttt{ResponseStatusException} mit dem Status \texttt{404 NOT FOUND}. 
    Dies ermöglicht dem Frontend eine klare Unterscheidung zwischen einer erfolgreichen leeren Suche und einem ungültigen Zugriff.
    \item \textbf{Kombinierte Suchfunktionen:} Alle Endpunkte unterstützen komplexe Filteroperationen. 
    Über Query-Parameter können Suchen nach exakten Namen, Namensbestandteilen (\texttt{nameContains}) oder Zugehörigkeiten (z. B. \texttt{departmentId}) kombiniert werden.
    Diese serverseitige Filterung erfolgt über dynamische \texttt{Specifications}. 
    \item \textbf{Validierungslogik:} Um inkonsistente Zustände zu vermeiden, implementiert das Backend eine Validierung der Parameterkombinationen. 
    So ist beispielsweise die gleichzeitige Verwendung von exakter Suche und Teilstring-Suche untersagt und führt zu einem \texttt{400 BAD REQUEST}.
    \item \textbf{Dynamische Spezifikationen und Paginierung:}
    Um die Abfragen auch bei großen Datenmengen performant zu halten, sind alle Listen-Endpunkte standardmäßig paginiert. 
    Dies wird durch die Integration von \texttt{Spring Data Pageable} realisiert. 
   

\end{itemize}

\subsubsection{Verarbeitung von Entitäten und Generierung von SQL-Statements durch das \texttt{Spring} - Framework}
Die Umwandlung von Klassen in SQL-Statements erfolgt in mehreren Schritten zur Laufzeit durch den \texttt{JPA}-Provider \texttt{Hibernate}, der vom \texttt{Spring}-Framework integriert wird.
Dabei dienen die Annotationen in der Entitätsklasse primär als Metadaten, die beschreiben, wie Objekte auf relationale Strukturen abgebildet werden sollen (siehe \ref{lst:bookingEntity}).
\\
Beim Start der Anwendung scannt das Framework alle Klassen, die mit \texttt{@Entity} annotiert sind. 
Diese werden als persistente Entitäten registriert. 
Ergänzende Annotationen wie \texttt{@Table}, \texttt{@Id} oder \texttt{@ManyToOne} liefern zusätzliche Informationen über Tabellenname, Primärschlüssel und Beziehungen.
\\
Aus diesen Informationen erstellt der \texttt{JPA}-Provider ein internes Metamodell, das die Struktur der Datenbank aus Sicht der Anwendung beschreibt.
Dabei werden unter anderem folgende Zuordnungen definiert:

\begin{itemize}
\item Klassen werden Tabellen zugeordnet (durch \texttt{@Entity} und \texttt{@Table})
\item Attribute werden Spalten zugeordnet (durch \texttt{@Column}, \texttt{@Temporal})
\item Beziehungen werden als Fremdschlüssel modelliert (durch \texttt{@ManyToOne} und \texttt{@JoinColumn})
\end{itemize}

Dieses Mapping existiert vollständig im Speicher und bildet die Grundlage für alle weiteren Datenbankoperationen.
\\
Zur Laufzeit verwaltet das Framework alle geladenen und zu speichernden Objekte in einem sogenannten Persistenzkontext. 
Dieser fungiert als Zwischenschicht zwischen Anwendung und Datenbank.
Wird eine Entität geladen oder gespeichert, wird sie in diesen Kontext aufgenommen. 
Änderungen an Objekten werden dabei automatisch erkannt, ohne dass explizit SQL formuliert werden muss.
\\
Erst bei konkreten Datenbankoperationen erzeugt das Framework die benötigten SQL-Statements. 
Dies geschieht dynamisch auf Basis des Mapping-Modells und des aktuellen Objektzustands.
\\So wird beispielsweise beim Speichern einer neuen Entität ein \texttt{INSERT}-Statement erzeugt, wobei Annotationen wie \texttt{@GeneratedValue} die Behandlung des Primärschlüssels beeinflussen.
Auch beim Abrufen und den generierten \texttt{SELECT}-Statements können Konfigurationen wie das verzögerte Laden von Beziehungen (durch \texttt{@ManyToOne} und \texttt{LAZY}) zur Netzwerkoptimierung mit einbezogen werden.

\newpage
\begin{lstlisting}[language=Kotlin, caption={Booking-Entität mit Spring Annotations in Kotlin}, label={lst:bookingEntity}]
@Entity
@Table(name = "bookings")
data class Booking(
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long? = null,

    @Temporal(TemporalType.DATE)
    @Column(name = "\"from\"")
    val from: Date,

    @Temporal(TemporalType.DATE)
    val until: Date?,

    @Convert(converter = BookingState.BookingStateConverter::class)
    val state: BookingState,

    @ManyToOne
    @JoinColumn(name = "room")
    val room: Room,

    @ManyToOne
    @JoinColumn(name = "patient")
    val patient: Patient
)

\end{lstlisting}


\subsubsection*{Fazit}
Die Kombination aus Kotlin und REST bietet eine robuste Grundlage für das Projekt. 
Während Kotlin die interne Datenintegrität und Typsicherheit garantiert, stellt REST eine flexible und standardisierte Kommunikation nach außen sicher,
wodurch das Hospital Management System sowohl sicher nutzbar als auch offen für zukünftige Erweiterungen gestaltet ist.
