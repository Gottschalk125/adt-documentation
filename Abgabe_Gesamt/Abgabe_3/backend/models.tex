\section{Datenbank-Optimierung}
Zur Steigerung der Abfrageperformance und Reduzierung der Systemlast bei steigenden Datenmengen wurden gezielte Optimierungen auf Datenbankebene durchgeführt. 

\subsection{ID-Typumstellung und Indexierung}
In der initialen Entwicklungsphase wurden Entitäten primär über UUIDs (Universally Unique Identifiers) identifiziert. 
Obwohl UUIDs Vorteile bei der dezentralen Generierung bieten, führen sie bei relationalen Datenbanken durch ihre Größe (128 Bit) und ihren unstrukturierten Aufbau zu einer Fragmentierung des Cluster-Indizes. 
Dies beeinträchtigt die Performance von \texttt{SELECT}- und Filteranfragen erheblich und verhindert die effiziente Nutzung von Indizes.

Aus diesem Grund wurden alle UUIDs als Primary Keys durch \texttt{Long} bzw. \texttt{BIGINT} auf Datenbank- und Backendebene umgestellt. Dies ermöglichte eine performantere Indexierung und beschleunigte Joins zwischen den Tabellen. 

Darüber hinaus wurden für Klassen, die einer hohen Frequenz an Filter- und Sortierabfragen unterliegen, dedizierte Indizes, teilweise als zusammengesetzte Composite Indizes, definiert.

\subsection{Funktionale Begründung der Indizes anhand von Use Cases}
Die gewählten Indizes wurden exakt auf die primären Geschäftsprozesse und Abfragemuster des Krankenhaus-Management-Systems abgestimmt. Im Folgenden wird die Notwendigkeit der Indizes anhand konkreter Use Cases der jeweiligen Fachbereiche begründet.

\subsubsection{Buchungssystem (Bookings)}
Das Belegungsmanagement ist eine der schreib- und leseintensivsten Komponenten. Hier müssen Konflikte vermieden und Verfügbarkeiten in Echtzeit geprüft werden.

\begin{itemize}
    \item \textbf{\texttt{idx\_bookings\_from\_id}:} Sinnvoll für chronologische Auflistungen aller Buchungen im System (z.\,B. \textit{„Zeige alle Buchungen ab dem Datum X“}), kombiniert mit der ID, um bei der Pagination (Keyset-Verfahren) ein stabiles und performantes Vorwärtsblättern zu ermöglichen.
    \item \textbf{\texttt{idx\_bookings\_patient\_from\_id}:} Unterstützt den Use Case der Patientenakte: Wenn ein Arzt die Historie eines spezifischen Patienten aufruft (\textit{„Welche Buchungen hatte Patient X in der Vergangenheit?“}), erlaubt dieser zusammengesetzte Index das sofortige Filtern nach der Patienten-ID und die anschließende chronologische Sortierung.
    \item \textbf{\texttt{idx\_bookings\_room\_from\_until}:} Kritisch für die Raumbelegungs- und Bettenplanung. Bei der Neuaufnahme eines Patienten muss das System prüfen, ob ein bestimmtes Zimmer in einem Zeitraum frei ist (\textit{„Ist Zimmer X vom Datum A bis B belegt?“}). Der Index verhindert rechenintensive Full-Table-Scans bei der Überschneidungsprüfung von Datumsbereichen.
    \item \textbf{\texttt{idx\_bookings\_state\_from\_id}:} Optimiert administrative Dashboards und Stationsübersichten. Das Pflegepersonal filtert hierbei nach dem aktuellen Status (z.\,B. \textit{„Zeige alle aktuell 'aktiven' oder 'ausstehenden' Buchungen“}) sortiert nach dem Startdatum.
\end{itemize}

\subsubsection{Patientenverwaltung (Patient)}
In der Patientenverwaltung steht neben der Performance vor allem die Datenqualität und das Verhindern von Dubletten im Vordergrund.

\begin{itemize}
    \item \textbf{\texttt{uq\_patient\_person} (Unique Constraint / Index):} Setzt die geschäftliche Validierungsregel um, dass eine physische Person (\texttt{person\_id}) im System nur exakt ein einziges Mal als \textit{Patient} registriert sein darf. Dies verhindert redundante Datenhaltung und logische Fehler bei der Zuweisung von Krankenakten.
\end{itemize}

\subsubsection{Medizinische Dokumentation (Diagnosis)}
Diagnosen bilden das datenintensive Rückgrat für medizinische Auswertungen, Visiten und die Abrechnung. Die Indizes unterstützen hier komplexe, filterbasierte Analysen.

\begin{itemize}
    \item \textbf{\texttt{idx\_diagnosis\_date\_id}:} Unterstützt globale, zeitbasierte Reportings der Klinikleitung (z.\,B. \textit{„Generiere den medizinischen Monatsbericht für Mai“}), gepaart mit der ID für die Cursor-Pagination.
    \item \texttt{idx\_diagnosis\_disease\_date}: Relevant für das epidemiologische Monitoring und Infektionsschutzberichte. Wenn das System statistisch auswerten muss, wie häufig eine bestimmte Krankheit (\texttt{disease}) in einem bestimmten Zeitraum aufgetreten ist, beschleunigt dieser Index die Aggregation massiv.
    \item \textbf{\texttt{idx\_diagnosis\_doctor\_date\_id}:} Verwendet für die Leistungsdokumentation und Visitenplanung von Ärzten (\textit{„Welche Diagnosen hat Arzt X heute oder in der letzten Woche gestellt?“}).
    \item \textbf{\texttt{idx\_diagnosis\_medication}:} Hilft bei der klinischen Entscheidungsunterstützung und Pharmakovigilanz (Arzneimittelsicherheit). Wenn untersucht werden muss, bei welchen Diagnosen ein bestimmtes Medikament verordnet wurde (oder umgekehrt Wechselwirkungen zu prüfen), erlaubt dieser Index einen schnellen Cross-Lookup.
    \item \textbf{\texttt{idx\_diagnosis\_patient\_date\_id}:} Der wichtigste Index für die tägliche Visite. Ruft ein Arzt die Akte eines Patienten auf, müssen alle bisherigen Diagnosen sofort und streng chronologisch (neueste zuerst) angezeigt werden. Dieser Index stellt sicher, dass diese Abfrage selbst bei Millionen von Diagnoseeinträgen in Millisekunden reagiert.
\end{itemize}


\subsection{Beispielhafte Implementierung im Backend}
Das folgende Codebeispiel zeigt die optimierte Datenklasse \texttt{Booking}. Hierbei wurden Indizes für häufig genutzte Filterkombinationen wie \texttt{room}, \texttt{patient} sowie die zeitliche Komponente \texttt{from} und \texttt{until} hinterlegt:

\begin{lstlisting}[style=kotlinstyle, caption={Optimierte Booking-Entität}]
@Entity
@Table(
    name = "bookings",
    indexes = [
        Index(name = "bookings_pkey", columnList = "id", unique = true),
        Index(name = "idx_bookings_from_id", columnList = "\"from\", id"),
        Index(name = "idx_bookings_patient_from_id", columnList = "patient, \"from\", id"),
        Index(name = "idx_bookings_room_from_until", columnList = "room, \"from\", until"),
        Index(name = "idx_bookings_state_from_id", columnList = "state, \"from\", id")
    ]
)
data class Booking(
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long? = null,

    @Temporal(TemporalType.DATE)
    @Column(name = "\"from\"")
    val from: Date,

    @Temporal(TemporalType.DATE)
    val until: Date?,

    @Convert(converter = BookingState.BookingStateConverter::class)
    val state: BookingState,

    @ManyToOne
    @JoinColumn(name = "room")
    val room: Room,

    @ManyToOne
    @JoinColumn(name = "patient")
    val patient: Patient
)
\end{lstlisting}
